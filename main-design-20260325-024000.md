## Problem Statement

Jason has a standard single-page portfolio site (HTML/CSS/JS) with placeholder images, a generic scrolling layout, and no memorable interaction model. He wants something experimental and weird that makes developer friends say "how did you build this?" The current site works but doesn't stand out from thousands of identical portfolio templates.

## What Makes This Cool

The portfolio IS the project. Instead of a passive page you scroll through, visitors explore a fully interactive Windows 98 desktop. Double-click icons, drag windows around, browse projects in a file explorer, view live deployed apps in an Internet Explorer window. The OS metaphor creates discovery and delight — people spend 5-10 minutes clicking around instead of 30 seconds scrolling past. And when they inspect the source? Vanilla JS, no framework. That's the second "whoa."

## Constraints

- Vanilla HTML/CSS/JS only (no React, no build step) — the simplicity is part of the flex
- Must work on modern browsers (Chrome, Firefox, Safari, Edge)
- 98.css library handles the Windows 98 visual foundation
- All current portfolio content (about, skills, 3 projects, contact info) must be preserved
- One app must be a browser window that embeds a real external website via iframe (with fallback — see iframe strategy below)
- Static site — deployable to GitHub Pages, Vercel, or Netlify with zero config
- Desktop-only experience. On mobile/tablet (viewport < 768px), show a full-screen Win98-style dialog box: "This program requires a desktop computer to run. Please visit on a larger screen." styled as a system error dialog. The OK button scrolls to a hidden `<section id="mobile-fallback">` within `index.html` that contains a plain, unstyled list of: bio text, skills list, project names with links, and contact email. This section is `display: none` on desktop viewports
- 98.css loaded via CDN (`unpkg.com/98.css`). Skill icons use inline SVGs from devicons CDN (`cdn.jsdelivr.net/gh/devicons/devicon`). Two CDN dependencies total, both well-established

## Premises

1. Full commitment to the Win98 desktop metaphor — no hybrid where some parts are Win98 and others are a normal website. Exception: one "Internet Explorer" window embeds a real external site.
2. The "apps" inside the desktop should be portfolio-relevant functional tools, not just generic Win98 nostalgia apps.
3. 98.css handles the visual foundation; custom vanilla JS handles window management and app logic. No heavy frameworks.

## Approaches Considered

### Approach A: 98.css + Custom Window Manager (CHOSEN)
Use 98.css for all visual styling, write a custom lightweight window manager in vanilla JS. Desktop icons open draggable/resizable windows. One window is an iframe "browser" embedding a real site. Full control, tiny bundle, maximum technical credibility.

- Effort: M (human: ~3 days / CC: ~1 hour for MVP, ~2 hours with stretch goals)
- Risk: Low
- Pros: Battle-tested 98.css, full control, zero deps beyond 98.css, tiny bundle
- Cons: Must write window manager (drag, resize with edge hit-testing, minimize/maximize, z-order stacking, taskbar sync — each non-trivial)

### Approach B: OS-GUI.js Full Stack
Use OS-GUI.js which provides both Win98 styling AND window management out of the box. Focus effort entirely on building the apps inside the windows.

- Effort: S (human: ~2 days / CC: ~20 min)
- Risk: Low
- Pros: Window management solved, faster to ship
- Cons: Less control, smaller library community

### Approach C: 98.css + React Component Architecture
Use 98.css for styling but build the desktop in React. Each "app" is a component.

- Effort: M (human: ~4 days / CC: ~40 min)
- Risk: Med
- Pros: Clean component model, React skills on display
- Cons: Heavier toolchain, less "wow this is vanilla JS" factor

## Recommended Approach

**Approach A: 98.css + Custom Window Manager.** Writing the window manager yourself in vanilla JS is the whole flex. Developer friends will inspect the source, see no framework, and that's the "how did you build this?" moment. The wireframe screenshot at `/tmp/gstack-sketch.png` shows the layout concept.

### Desktop Architecture

**Desktop Icons (left side):**
- About Me — opens a "Properties" style dialog with avatar and bio
- Skills — opens a window with skill icons in a grid or tree view
- My Projects — opens a File Explorer window (`C:\Users\Jason\Projects`)
- Internet Explorer — opens a browser window with address bar and iframe
- Contact Me — opens a "New Message" email-style window with form fields
- Resume.pdf — opens a document viewer window displaying the resume (or triggers download if no PDF is provided yet — placeholder behavior until Jason supplies a real PDF)

**Window Manager Features:**
- Drag windows by title bar (mousedown on title bar starts drag, mousemove updates position, mouseup ends)
- Resize windows by 8px hit-zone inset from the window's outer border. Corner zones (16x16px at each corner) take priority over edge zones when overlapping. Minimum window size: 200x150px. Cursors: `nw-resize`, `ne-resize`, `sw-resize`, `se-resize` for corners; `n-resize`, `s-resize`, `e-resize`, `w-resize` for edges. App content inside windows must use `%` or `min-content` widths (no fixed `px` layouts) so they reflow at any window size. **Note: resize is the highest-complexity window manager feature. If it proves time-consuming, ship MVP without resize and add it as a follow-up polish pass.**
- Minimize to taskbar (window hides, taskbar button stays), maximize to full viewport minus taskbar height (36px), close removes window and taskbar button
- Z-order: clicking anywhere inside a window brings it to front. Implementation: maintain a global counter starting at `z-index: 100`, increment on each focus. Taskbar is fixed at `z-index: 9999` (always on top). Counter never needs resetting for a portfolio use case (would take millions of clicks to hit browser limits). Taskbar button click toggles minimize/restore.
- Double-click desktop icon to open, single-click to select (highlight with dotted border)

**Taskbar:**
- Start button (left) — present in MVP but **non-functional** (clicking does nothing). Start Menu behavior is stretch goal #1
- Open window buttons (middle) — click to minimize/restore
- System tray clock (right) — shows current time

**Apps:**

1. **About Me (Properties dialog):** System Properties-style window with avatar image and bio text. Single view — no tabs (keep it focused on the bio, not artificial content padding).

2. **Skills (Control Panel-style grid):** Grid of skill icons with labels, styled like Win98 Control Panel. Icons loaded as `<img src="https://cdn.jsdelivr.net/gh/devicons/devicon@latest/icons/...">` tags (simplest approach — no need for inline SVG or CSS fill control in a skills grid). 14 skills from current site (HTML, CSS, JS, TS, Python, React, Next.js, Tailwind, MongoDB, SQL, AWS, Git, GitHub, Claude Code).

3. **My Projects (File Explorer):** Explorer chrome with address bar showing `C:\Users\Jason\Projects`. Toolbar labels (File, Edit, View, Help) are **decorative only** — they exist for visual authenticity but are not functional menus. Folder icons for each project. **Double-clicking a project folder opens the IE window and loads that project's URL in the iframe.** This is the single, committed interaction path (no ambiguity).

4. **Internet Explorer:** Address bar (editable), Back/Forward/Refresh buttons, iframe viewport. Projects launch here from the Explorer. **Iframe fallback strategy:** Set a 3-second timeout after calling `iframe.src = url`. Listen for the `load` event — if it fires, attempt `iframe.contentWindow.document` access in a try/catch. If the load event never fires within the timeout, OR if `iframe.contentWindow.document` access throws a `SecurityError` (indicating cross-origin block), show a Win98-style "The page cannot be displayed" error inside the viewport with an "Open in new window" link (`window.open(url, '_blank')`). This is a best-effort heuristic — `iframe.onerror` does NOT fire for X-Frame-Options/CSP blocks in modern browsers. Acknowledge that some sites may load but render blank; the fallback link ensures users can always access the project.

5. **Contact Me (Outlook Express-style):** Email composition form — To (pre-filled with jason.zeng@pursuit.org, displayed as plain text so users can copy it manually), Subject, Message body, Send button. **Send triggers a `mailto:` link** (zero backend). Below the Send button, display the email address as a clickable/copyable fallback for users without a mail client configured. If Jason later wants a form service, swap the mailto for a POST to Formspree.

6. **Resume.pdf:** Opens a document viewer window with `<iframe src="assets/resume.pdf">` (leverages browser's built-in PDF viewer). Below the iframe, always show a "Download PDF" link as fallback for browsers without PDF support. Until Jason provides a real PDF, show a Win98 "File not found" error dialog instead.

### Technical Implementation

```
personal-portfolio/
  index.html          # Desktop shell, icons, taskbar
  css/
    # 98.css loaded via CDN (unpkg.com/98.css) — not vendored locally
    desktop.css       # Custom desktop layout, icon grid, taskbar
    apps.css          # Per-app styling overrides
  js/
    window-manager.js # Drag, resize, minimize, maximize, close, z-order
    desktop.js        # Icon interactions, Start menu, app launcher
    apps/
      about.js        # About Me window logic
      skills.js       # Skills display
      explorer.js     # File Explorer for projects
      browser.js      # IE iframe wrapper
      contact.js      # Contact form
      resume.js       # Resume PDF viewer / placeholder
  assets/
    icons/            # Desktop icons (custom 32x32 PNGs or inline SVGs for Win98 feel)
```

### MVP vs. Stretch Goals

**MVP (must ship):**
- Desktop with 6 icons, taskbar with Start button + clock
- Window manager: drag, minimize/maximize/close, z-order (resize is deferred to post-MVP polish — see note in Window Manager Features)
- All 6 app windows with content
- IE iframe with fallback strategy
- Mobile gate (system error dialog on small screens)

**Stretch goals (post-MVP, in priority order):**
1. Start Menu — click Start to see a menu with all apps and "Shut Down" option
2. Window snapping — drag to screen edge to snap half-screen
3. Right-click context menu — decorative only ("New > Folder", "Properties", "Refresh" do nothing but look authentic)
4. Sound effects — startup chime, error beep, window open/close sounds
5. Boot sequence — brief Win98 boot animation (low priority: delays content discovery, which works against the stated success criterion of exploration time)
6. Basic keyboard accessibility — Tab to cycle between open windows, Escape to close focused window, Enter to activate focused desktop icon
7. Additional functional apps Jason wants (to be determined — see Open Questions)

**Known limitations:**
- No full keyboard/screen-reader accessibility for window dragging/resizing. The mobile fallback section provides accessible content for all users. This is acceptable for a fun side project but would need addressing for a production app.
